# TICKET-049: 行业比较接口逐股抓取财务数据导致响应慢

**优先级:** P3 - 低
**状态:** 📋 待开始
**依赖:** -
**预计工时:** 1d

## 症状

`src/zhanfa/api/services/stock_service.py` 的 `get_industry_comparison()` 遍历行业成分股，并逐只调用 `fetcher.financial(code)`：

```py
for _, row in stocks.iterrows():
    fin = fetcher.financial(code)
```

行业股票数量较多时，接口会串行读取或抓取大量财务数据。若缓存缺失，真实 API 请求可能长时间阻塞。

## 根因分析

行业比较接口缺少批量财务摘要层，也没有限制比较对象数量、超时、缓存策略或后台预计算。当前实现更像离线分析流程，不适合作为同步 HTTP 请求直接暴露。

## 修复方案

### 1. 为同步接口设置边界

- 增加 `limit` 参数或默认只取前 N 个样本
- 缓存缺失时优先返回空指标/待刷新状态，不在请求内抓取大量远端数据
- 给返回值增加数据新鲜度或缺失原因

### 2. 复用缓存与批量读取

优先从本地缓存或数据库读取最新财务摘要。必要时新增批量 helper，避免每只股票独立初始化读取流程。

### 3. 考虑后台刷新

若用户需要完整行业比较，可改为：

- 前端触发后台刷新任务
- `/api/scheduler/trigger` 或新端点刷新行业财务缓存
- 行业比较接口只读已缓存结果

## 验收标准

- [ ] 行业比较接口不会因大量缓存缺失而串行触发无限制远端抓取
- [ ] 响应时间与行业成分股数量有明确上限
- [ ] 前端能区分“指标为 0”和“数据缺失/未刷新”
- [ ] `uv run pytest tests/test_api tests/test_services -q` 通过
