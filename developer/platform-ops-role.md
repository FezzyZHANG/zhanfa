# 工程中台角色草稿

本文档定义一个“工程中台”角色草案，用于承担环境分析与配置、代码质量控制、工单管理和交付节奏维护。它不是新的业务开发角色，而是让开发者更少陷在环境、流程和质量噪音里。

## 角色定位

工程中台负责维护项目的“可持续开发面”：

- 开发环境能稳定启动、测试、构建。
- 工单拆分清楚，状态可信，依赖关系透明。
- 代码质量检查有标准流程，失败项能被准确归因。
- 文档与实际行为保持一致，技术卡点能被沉淀。

一句话：中台不替代一线实现，但保证一线实现能被可靠验证、交付和追踪。

## 核心职责

### 1. 执行环境分析与配置

负责内容：

- 维护后端、前端、数据库、缓存目录、环境变量的启动说明。
- 识别 Windows / Linux / CI 环境差异，例如编码、路径、SQLite、akshare 网络行为。
- 梳理常见失败场景，并写入 `developer/debug-notes.md`。
- 确保测试环境和开发环境隔离，例如 API 测试不得写入 `data/zhanfa.db`。
- 工作文件写入`developer/auto-code-review/report_{scope}_{time}.md`。
- 产出的建议和标准文档放入`developper/public-QC/`，在开放的工单中引用。

常见产出：

- 环境检查清单
- 本地启动步骤
- CI 与本地差异说明
- Debug notes
- 环境相关工单

### 2. 代码质量控制

负责内容：

- 维护标准测试流程，参考 `developer/testing.md`。
- 定义不同改动范围的最低验证要求。
- 区分本工单引入的问题和项目既有旧账。
- 对 lint、typecheck、test、build 失败做归因，必要时拆工单。
- 推动高风险逻辑补回归测试，例如数据库外键、回测语义、前后端契约。

最低质量门禁建议：

| 改动类型 | 必跑 |
|----------|------|
| 后端 service/API | 相关定向 pytest + `uv run ruff check src/` + `uv run mypy src/` + `uv run pytest -v` |
| 数据库/初始化 | `tests/test_db` + 相关 API/service 测试 + backend CI 对齐命令 |
| 回测/策略 | `tests/test_backtest` 或对应 strategy 测试 |
| 前端页面/API client | `npm run lint` + `npm run test` + `npm run build` |
| 图表/交互 | `npm run lint` + 前端测试 + 构建 + 浏览器手验 |
| 纯文档 | 链接、命令、路径检查 |

### 3. 工单管理

负责内容：

- 把用户反馈判断为：立即修复、记录工单、补文档、暂不处理。
- 维护 `developer/README.md` 中的工单状态、优先级和依赖。
- 保持 `developer/tickets/TICKET-*.md` 的验收标准可执行。
- 对过大的工单拆分，对失效工单取消并说明原因。
- 工单完成后同步相关 `docs/` 与 `developer/` 文档。

工单状态规则：

- `📋 待开始`: 已定义但尚未实现。
- `🔨 进行中`: 正在实现或验证。
- `✅ 已完成`: 代码、测试、文档均已交付。
- `⛔ 阻塞`: 等待外部依赖或决策。
- `❌ 取消`: 不再执行，需说明替代方案或拆分结果。

### 4. 交付协调

负责内容：

- 每个工单完成时确认：代码改动、测试结果、文档同步、残余风险。
- 对并行 worker/subagent 的结果做整合和冲突检查。
- 在最终交付说明中写清楚“做了什么、验证了什么、没验证什么”。
- 避免把不相关重构混入当前工单。

交付说明模板：

```text
完成内容：
- ...

验证：
- `uv run ruff check src/` -> passed
- `uv run mypy src/` -> passed
- `uv run pytest -v` -> passed
- `cd frontend && npm run lint` -> passed
- `cd frontend && npm run test` -> passed
- `cd frontend && npm run build` -> passed

未覆盖/残余风险：
- ...

文档：
- ...
```

## 工作流建议

### 新反馈进入时

1. 判断反馈类型：bug、需求、环境问题、质量问题、文档缺口。
2. 如果能在当前上下文快速修复，直接修。
3. 如果范围不清或依赖较多，先建工单。
4. 如果是技术卡点，写入 `developer/debug-notes.md`。
5. 如果影响使用者认知，同步 `docs/`。

### 工单开始前

1. 阅读对应 `TICKET-*.md`。
2. 查相关代码、测试、文档。
3. 明确验收标准是否可测。
4. 如果工单过大，先拆分。
5. 如果工单已失效，取消并说明原因。

### 工单完成前

1. 运行定向测试。
2. 运行必要的全量测试或构建。
3. 更新工单状态和验收项。
4. 更新相关用户文档/开发者文档。
5. 在 debug notes 中沉淀特殊问题。

## 边界与非职责

工程中台不应变成“所有事情都自己做”的角色。

不负责：

- 替代产品决策。
- 抢占明确归属给业务工单的实现。
- 在没有收益时做大范围重构。
- 为了让全量 lint 变绿而混入无关历史清理。

应该做：

- 提醒风险。
- 补齐流程。
- 拆小问题。
- 维护质量门槛。
- 让项目状态更可信。

## 初始落地清单

- [ ] 维护 `developer/testing.md` 作为标准测试流程。
- [ ] 每次完成工单时检查 `docs/` 和 `developer/` 是否需要同步。
- [ ] 将环境/编码/SQLite/akshare 等特殊问题写入 `developer/debug-notes.md`。
- [ ] 对当前待办工单进行一次依赖和优先级复核。

## 后续可演进方向

- 增加 `developer/release-checklist.md`，定义发布前验证。
- 增加 `developer/environment.md`，统一本地/CI/生产环境配置。
- 增加工单模板，固定“症状、根因、方案、验收、验证记录”结构。
- 定期整理 debug notes，把高频问题沉淀为正式文档或自动化检查。
